# 需求文档

## 引言

Git Dashboard 是将现有 CodeStat Shell 脚本升级为 Web 应用面板的功能改造项目。CodeStat 目前以命令行方式批量统计多个 Git 仓库的提交记录，本次改造目标是将统计能力扩展为可视化的 Web 面板，支持跨平台自动发现 Git 仓库、智能识别项目边界，并以独立卡片形式展示每个项目的提交量。

**技术约束：** 使用 Go 语言编译为单个二进制文件部署，SQLite 作为本地数据存储，启动后自动打开浏览器访问本地面板（桌面应用体验）。

## 术语表

| 术语 | 定义 |
|------|------|
| **扫描根目录** | 用户配置的顶层目录列表，系统在这些目录下递归查找 Git 仓库 |
| **Git 仓库** | 包含 `.git` 目录的文件夹，是一个可以被 git 管理的代码仓库 |
| **项目** | 面板中展示的基本单元，可能对应单个 Git 仓库，也可能对应包含多个 Git 仓库的上级目录（如 monorepo） |
| **项目目录级别** | 用户可调整的，决定项目以哪一层目录作为面板展示边界（例如"每个 Git 仓库独立展示"或"将某上级目录作为一个项目展示"） |
| **提交量** | 指定日期范围内，代码的新增行数、删除行数、文件变更数及净增行数 |

## 需求

### Requirement 1: 扫描范围配置

**User Story:** 作为开发者，我想要设置 Git 仓库的扫描范围目录，以便系统只在特定磁盘或文件夹中查找项目。

#### Acceptance Criteria

1. WHEN 用户首次启动应用，该系统 SHALL 使用平台默认扫描规则自动设定扫描范围。
2. WHEN 运行在 Windows 平台，默认扫描范围 SHALL 包含除系统盘（通常为 C:）以外的所有可用磁盘根目录。
3. WHEN 运行在 macOS 平台，默认扫描范围 SHALL 包含当前用户的 HOME 目录（/Users/用户名）。
4. WHEN 运行在 Linux 平台，默认扫描范围 SHALL 包含当前用户的 HOME 目录。
5. WHEN 用户在设置界面修改扫描范围后，该系统 SHALL 在下次扫描时应用新的范围配置。
6. WHEN 扫描范围配置发生变更，该系统 SHALL 持久化配置到本地存储。

---

### Requirement 2: Git 仓库自动发现

**User Story:** 作为开发者，我想要系统自动发现扫描范围内的所有 Git 仓库，以便无需手动逐个添加项目。

#### Acceptance Criteria

1. WHEN 系统执行扫描任务，该系统 SHALL 递归遍历所有已配置的扫描根目录以发现包含 `.git` 子目录的文件夹。
2. WHEN 扫描发现 `.git` 目录，该系统 SHALL 将该文件夹标记为 Git 仓库并记录其绝对路径。
3. IF 扫描根目录内的子目录数量超过 10000 个，该系统 SHALL 限制扫描深度最大为 5 层以避免性能问题。
4. WHEN 扫描完成，该系统 SHALL 在面板中展示所有已发现的 Git 仓库列表。

---

### Requirement 3: 智能项目分组

**User Story:** 作为开发者，我想要系统自动识别项目边界，将属于同一项目的多个 Git 仓库合理分组展示。

#### Acceptance Criteria

1. WHEN 扫描发现多个 Git 仓库存在于同一个上级目录下（如 monorepo 结构），该系统 SHALL 默认将这些仓库分组展示，以上级目录名作为项目名称。
2. WHEN 上级目录本身也是一个 Git 仓库且其子目录中存在其他 Git 仓库，该系统 SHALL 将上级仓库与其子仓库分开展示为独立项目。
3. WHEN 一个目录下仅有一个 Git 仓库，该系统 SHALL 以该仓库所在的直接父目录名作为项目名称。

---

### Requirement 4: 项目目录级别调整

**User Story:** 作为开发者，我想要手动调整某个项目的目录级别，以便修正系统自动分组的结果。

#### Acceptance Criteria

1. WHEN 用户点击项目的"调整级别"按钮，该系统 SHALL 提供向上或向下调整目录层级的选项。
2. WHEN 用户向上调整项目层级，该系统 SHALL 将当前项目的展示边界扩大到上级目录，将该上级目录下的其他 Git 仓库合并入同一个项目。
3. WHEN 用户向下调整项目层级，该系统 SHALL 将当前项目拆分为以子目录为单位的独立项目。
4. WHEN 目录级别调整完成，该系统 SHALL 持久化调整结果并在后续扫描中保持用户的调整。

---

### Requirement 5: 提交量统计与展示

**User Story:** 作为开发者，我想要查看每个项目在指定日期范围内的提交量统计。

#### Acceptance Criteria

1. WHEN 面板展示项目列表，该系统 SHALL 为每个项目显示独立卡片，包含项目名称和最近统计的提交量数据。
2. WHEN 用户选择统计日期范围（昨天/今天/自定义日期），该系统 SHALL 重新统计所选日期范围内各项目的提交量并更新卡片展示。
3. WHEN 统计某个项目时，该系统 SHALL 区分当前用户与其他用户的贡献，显示个人新增行数、删除行数、文件变更数和净增行数。
4. IF 指定日期为工作日且个人新增行数低于配置的标准值，该系统 SHALL 在卡片上展示未达标警告标识。
5. WHEN 统计完成，该系统 SHALL 在页面顶部展示所有项目的汇总信息，包括仓库总数、总新增行数、总删除行数和个人贡献占比。

---

### Requirement 6: Web 面板界面

**User Story:** 作为开发者，我想要通过浏览器访问一个 Web 面板来查看和管理所有项目的提交统计。

#### Acceptance Criteria

1. WHEN 用户启动应用，该系统 SHALL 启动 Web 服务器并在终端输出可访问的地址。
2. WHEN 用户访问面板首页，该系统 SHALL 展示项目卡片网格布局，每个卡片显示项目名称和最近提交量摘要。
3. WHEN 用户点击某个项目卡片，该系统 SHALL 展开该项目的详细统计信息，包含按日期展示的提交量折线图。
4. WHEN 页面顶部导航栏存在，该系统 SHALL 提供日期选择器和设置入口。
5. IF 使用移动设备访问面板，该系统 SHALL 自适应布局为单列卡片展示。

---

### Requirement 7: 配置持久化

**User Story:** 作为开发者，我想要所有配置（扫描范围、项目级别、代码量标准）在应用重启后依然有效。

#### Acceptance Criteria

1. WHEN 用户修改任意配置项，该系统 SHALL 自动保存配置到本地持久化存储。
2. WHEN 应用重新启动，该系统 SHALL 从持久化存储加载上一次的配置。
3. IF 持久化存储损坏或不存在，该系统 SHALL 回退到默认配置并提示用户重新设置。

---

### Requirement 8: 跨平台支持

**User Story:** 作为开发者，我想要在同一套代码上运行面板，覆盖 Windows、macOS 和 Linux 系统。

#### Acceptance Criteria

1. WHEN 应用在 Windows 系统启动，该系统 SHALL 正确识别盘符并排除系统盘。
2. WHEN 应用在 macOS 系统启动，该系统 SHALL 使用 /Users 目录作为默认扫描根。
3. WHEN 应用在 Linux 系统启动，该系统 SHALL 使用 HOME 环境变量作为默认扫描根。
4. WHEN 系统执行 Git 命令，该系统 SHALL 根据操作系统使用正确的 Shell 执行方式。
